### 正文
```
We built aicfg — a zero-dependency CLI that generates CLAUDE.md and 
.cursorrules for AI coding agents. 

It detects your stack (Go, Next.js, Node/Express, Python/FastAPI) and 
generates production-grade rules covering naming, immutability, error 
handling, security, and testing — each with a one-line "Why" that 
explains the reasoning to the AI.

The dogfooding angle: aicfg configured itself using `aicfg init`. The 
CLAUDE.md in our repo is the output. Eat your own dogfood or don't ship.

The part we're genuinely curious about: we built an x402 (HTTP 402 
Payment Required) payment flow using Arbitrum USDC — no signup, no API 
keys, no email. Send USDC, verify on-chain, unlock premium stacks. Our 
AI agent completed this payment autonomously. 

We're not sure if developers will pay for AI agent configs with crypto. 
But we wanted to test the hypothesis that an AI agent can be its own 
merchant. That's the real experiment here.

Free tier: 4 stacks. Premium: 6 stacks at 10 USDC (Arbitrum).

Install: npm install -g github:ipythoning/aicfg
Repo: https://github.com/ipythoning/aicfg

Not asking for stars. Asking for feedback on the core assumption: 
do you actually want better AI agent configs, or do you just tolerate 
the current "write clean code" placeholder files?
```

## P2 — Cursor Forum / Claude Code Discussions

### 策略
不主动开贴。在别人的问题里提供价值。

### 监控关键词
- "Claude Code not following my rules"
- "CLAUDE.md not working"
- "AI keeps refactoring my code"
- "Cursor rules file not respected"

### 回复模板
```
Most people's AI config files are too vague. "Write clean code" 
isn't a rule — the AI doesn't know what "clean" means to your team.

What actually works: specific rules with justified reasoning. 
For example, instead of "handle errors", write "Every async 
route must forward errors to next(err) — because Express 
doesn't catch unhandled promise rejections and will crash."

We built a tool that generates these production-grade rules 
automatically based on stack detection. Might help:
https://github.com/ipythoning/aicfg

(Disclaimer: I built it. But the config patterns are the real 
insight, not the tool.)
```
